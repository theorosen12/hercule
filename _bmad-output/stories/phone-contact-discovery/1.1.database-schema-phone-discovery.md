# Story 1.1: Database Schema for Phone Discovery

## Status

Done

## Story

**As a** backend developer,
**I want** database models to store user phone hashes,
**so that** users can be found by others who have their phone number.

## Acceptance Criteria

1. Create Prisma schema with `HashedPhoneNumber` model (id, userId, hashedPhone, timestamps) - stores user's OWN phone hash
2. Add relation from User to HashedPhoneNumber with cascade delete
3. Create database indexes on `hashedPhone` and `userId` fields
4. Run `prisma migrate dev` successfully
5. Ensure unique constraint on `hashedPhone` field

### Key Implementation Details

**Prisma Schema:**
```prisma
model User {
  id                    String              @id @default(uuid())
  // ... existing fields ...

  // Phone Discovery (NEW)
  hashedPhoneNumber     HashedPhoneNumber?

  // ... existing relations ...
}

model HashedPhoneNumber {
  id          String   @id @default(uuid())
  userId      String   @unique
  hashedPhone String   @unique
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt

  user        User     @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@index([hashedPhone])
}
```

**GraphQL User Type Extension:**
```typescript
@ObjectType()
export class User {
  // ... existing fields ...

  @Field(() => Boolean)
  hasPhoneNumber: boolean; // Computed from hashedPhoneNumber relation
}
```

**Field Resolver for hasPhoneNumber:**
```typescript
@ResolveField(() => Boolean)
async hasPhoneNumber(@Parent() user: User): Promise<boolean> {
  const hash = await this.prisma.hashedPhoneNumber.findUnique({
    where: { userId: user.id },
  });
  return hash !== null;
}
```

### Design Decisions

| Decision | Rationale |
|----------|-----------|
| One phone per user (`userId @unique`) | PRD specifies single phone number per user |
| `hashedPhone @unique` | Prevents same phone registered by multiple users |
| `onDelete: Cascade` | Clean up phone hash when user account is deleted |
