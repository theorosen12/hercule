# Story 4.2: Backend - Return friendRequest Field in getContactSuggestions

## Status

Draft

## Story

**As a** mobile client consuming the contact suggestions query,
**I want** the `getContactSuggestions` query to include the `friendRequest` field with its status for each suggested user,
**So that** I can display the correct friend request state (none, pending, accepted) without maintaining local state on the client.

## Acceptance Criteria

1. `getContactSuggestions` query response includes a `friendRequest` field for each suggestion
2. `friendRequest` returns `null` if no friend request exists between the current user and the suggested user
3. `friendRequest` returns `{ id, status }` if a friend request exists (in either direction)
4. Status values cover: `PENDING`, `ACCEPTED`, `DECLINED` (or whatever the existing enum is)
5. The field considers friend requests in **both directions** (sent by current user OR received from suggested user)
6. Query performance remains acceptable (no N+1 problem - use dataloader or join)
7. The `friendRequest` type should be a proper GraphQL type with `__typename` for Apollo cache normalization

## Tasks / Subtasks

- [ ] **Task 1: Add friendRequest Field to ContactSuggestion Type** (AC: 1, 7)
  - [ ] Add `friendRequest` field to the ContactSuggestion GraphQL type
  - [ ] Define return type: `FriendRequest` (nullable) with `id` and `status`
  - [ ] Ensure `FriendRequest` has proper `__typename` for Apollo normalization

- [ ] **Task 2: Resolve friendRequest for Each Suggestion** (AC: 2, 3, 4, 5)

  - [ ] OR where suggested user is sender AND current user is receiver

  - [ ] Alternatively, use a single JOIN when fetching suggestions

  - [ ] Update schema if using schema-first approach

  - [ ] Update mobile query definition to include new field

### Backend Resolution Logic

```typescript
// Pseudo-code for resolver
@FieldResolver(() => FriendRequest, { nullable: true })
async friendRequest(
  @Root() suggestion: ContactSuggestion,
  @Ctx() ctx: Context
): Promise<FriendRequest | null> {
  const currentUserId = ctx.user.id;
  const suggestedUserId = suggestion.id;

  // Find friend request in either direction
  return await friendRequestRepository.findOne({
    where: [
      { senderId: currentUserId, receiverId: suggestedUserId },
      { senderId: suggestedUserId, receiverId: currentUserId },
    ],
  });
}
```

### Performance Consideration

To avoid N+1:
```typescript
// Use DataLoader pattern
const friendRequestLoader = new DataLoader(async (userIdPairs) => {
  // Batch query all friend requests for the pairs
  const requests = await friendRequestRepository.find({
    where: userIdPairs.map(([currentId, suggestedId]) => [
      { senderId: currentId, receiverId: suggestedId },
      { senderId: suggestedId, receiverId: currentId },
    ]).flat(),
  });
  // Map back to input order
  return userIdPairs.map(([currentId, suggestedId]) =>
    requests.find(r =>
      (r.senderId === currentId && r.receiverId === suggestedId) ||
      (r.senderId === suggestedId && r.receiverId === currentId)
    ) ?? null
  );
});
```

### Apollo Cache Normalization

The `FriendRequest` type with `id` and `__typename` allows Apollo to normalize it in the cache. When `sendFriendRequest` mutation returns a `FriendRequest` with the same `id`, Apollo will automatically update the cached data. This is critical for Story 4.4.
